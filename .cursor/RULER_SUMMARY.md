# Cursor Ruler 調整 — 結論與摘要（care_rag_api）

## 目的

在保留 **SSOT、ECR、測試隔離、歷史註解** 的前提下，避免「凡改邏輯都要完整 ECR」導致 **已知檢索假陽性** 等 bug 無法務實修復；並區分 **無人批次** 與 **對話內明示任務**。

## 採行方案（精簡）

1. **變更分級 T0–T3**：小改直接做；缺陷用 T1；新行為用 T2；設計衝突／契約／安全用 T3。
2. **T1 預授權清單**（僅限 RAG/Graph/檢索類 **缺陷**）：假陽性、誤導分數、無來源硬答 → 可修但 **必須**附驗證。
3. **SSOT 缺口**：先用 ADR 或 `dev_readme` 補「預設行為」再動程式。
4. **批次自動化** 仍嚴格；**對話任務** 依分級執行。

## 優點（Pros）

- 界線清楚，**修 bug 不必每次都開完整 ECR**。
- 與「重大變更要批准」相容，只是 **細分執行層級**。
- 可追溯：PR 可標註 T1/T2/T3。
- 針對實際案例（如 `Organization` 撞 `type`、固定 `0.85`）給出 **白名單式**出口。

## 缺點／風險（Cons）

- T1/T2 **邊界需共識**，否則易爭議。
- T1 清單若過寬，可能 **繞過** 應有的產品／架構審查 → 應 **季檢** 清單。
- 依賴 **測試與驗證步驟**，否則易迴歸。
- ADR/`dev_readme` 增多時須約定 **與主設計的合併節奏**，避免 SSOT 分裂。
